# ТЗ: ИИ-агент управления рекламой в Яндекс.Директе

**Проект:** MYCHEAGENT / Студия ЧЕ
**Версия:** 1.0 (черновик от 2026-04-23)
**Владелец:** Денис Видяков

---

## 1. Цель

Автоматический ИИ-агент, который ежедневно анализирует рекламные кампании в Яндекс.Директе, формирует рекомендации по их улучшению, безопасно применяет рутинные изменения с подтверждением владельца и проводит A/B-тесты креативов. Сокращает время владельца на работу с Директом до 5–10 минут в день (просмотр сводки + один клик «применить»).

## 2. Контекст

- Аккаунт Директа: `vidyakovd88`, продукт «Студия ЧЕ» (онлайн-курсы по видеомонтажу и AI).
- Инфраструктура: n8n 2.15 на VPS, Telegram-бот Zeus как UI владельца, PWA `mycheagent.ru`, Claude Sonnet 4.6 / Haiku через AWstore.

  - [`20-yadirect-daily.json`](../20-yadirect-daily.json) — ежедневный отчёт + рекомендации.
  - [`01-videomontazh.md`](01-videomontazh.md) — бриф первой кампании.

## 3. Архитектура

### 3.1 Компоненты

| Компонент | Где | Назначение |
|---|---|---|
| Direct Proxy | `19-yandex-direct.json` | Тонкий REST-прокси над Direct API v5: campaigns/adgroups/ads/keywords/reports |
| Daily Analyzer | `20-yadirect-daily.json` | Cron 9:00 ЕКБ: отчёт за вчера → LLM-анализ → TG-сводка с рекомендациями |
| Recommendation Applier | в `01-orchestrator.json` (handler callback `yd_apply:*`) | По кнопке «Применить» проигрывает рекомендации через прокси |
| Weekly Report | новый воркфлоу `21-yadirect-weekly.json` | Еженедельная сводка по понедельникам, тренды, A/B-победители |
| A/B Tester | новый воркфлоу `22-yadirect-abtest.json` | Раз в 7 дней анализирует ротацию объявлений, отключает слабые, генерит замены через LLM |
| Trigger Watcher | новый воркфлоу `23-yadirect-watcher.json` | Раз в 30 минут проверяет аномалии (резкий перерасход, обвал CTR), шлёт алерт |
| Metrika Sync | новый воркфлоу `24-metrika-sync.json` | Тянет цели из Яндекс.Метрики, сводит с расходами Директа в CPA |

```
                                       │

                                       │
                                       ▼

                                       │
                                       ▼

```

### 3.3 Уровни автономности

| Уровень | Что делает | Подтверждение |
|---|---|---|
| **L0 — наблюдение** | Сбор статистики, отчёты, алерты | не требуется |
| **L1 — рекомендации** | Анализ, формирование изменений, отправка в TG | не требуется |
| **L2 — авто с подтверждением** | Применение L1-рекомендаций по кнопке | один клик |
| **L3 — авто без подтверждения, в узких рамках** | Минус-слова из «мусорных» запросов, мягкая корректировка ставок ±10% | не требуется (внутри лимитов) |

L3 включается **только** для определённых типов изменений (см. §5.4). Все остальные изменения — только L2 или ручные.

## 4. Функциональные требования

### 4.1 Ежедневный отчёт (готов на 80%)

- Триггер: cron `0 9 * * *` (ЕКБ).
- Источники: Direct `CAMPAIGN_PERFORMANCE_REPORT` + Метрика «Цели за вчера» + сравнение с позавчерашним днём.
- Формат TG-сообщения:
  ```
  📊 Директ-отчёт за 23 апреля

  Итого: показы 423 · клики 11 · CTR 2.6% · CPC 38₽ · расход 421₽ · заявки 1 · CPA 421₽

  По кампаниям:
  • Видеомонтаж — 11 кликов / 421₽ / 1 заявка ✅

  💬 Резюме: первый день — отлично, CTR в 3 раза выше нормы…

  🔧 Предлагаю:
  1. Поднять дневной бюджет до 500₽ — заявки идут
  2. Добавить минус-слово «бесплатно» — 2 пустых клика
  ```
- Кнопки: `✅ Применить (N)` · `📊 Детали` · `🚫 Игнор`.
- При нулевом расходе — короткое сообщение «вчера показов не было», без LLM.

### 4.2 Применение рекомендаций (нужно сделать)

При нажатии `✅ Применить`:
1. Достать `pendingRecs[date]` из static data.
2. Для каждой рекомендации вызвать прокси `/webhook/yadirect` с нужным action.
3. Поддерживаемые типы:
   - `pause_campaign` → `campaigns.suspend`
   - `resume_campaign` → `campaigns.resume`
   - `raise_budget` / `lower_budget` → `campaigns.update` (с проверкой ±50%)
   - `lower_cpc` / `raise_cpc` → `campaigns.update` (стратегия)
   - `add_negative_keyword` → `keywords.update` с `NegativeKeywords`
   - `pause_ad` → `ads.suspend`
   - `replace_creative` → НЕ применять автоматически, переслать LLM-задание в TG для генерации новых вариантов (см. 4.4)
4. После применения — отчёт в TG: «Применил 2 из 3, ошибки: replace_creative требует одобрения текстов».

### 4.3 Еженедельный отчёт (нужно сделать)

- Триггер: cron `0 9 * * 1` (понедельник 9:00 ЕКБ).
- Период: последние 7 дней vs предыдущие 7 дней.
- Содержание:
  - Динамика расхода / CTR / CPA.
  - Топ-3 объявления по CTR, топ-3 по CPA.
  - A/B-победитель за неделю.
  - Прогноз на следующую неделю (по тренду).
  - Список «закрытых вопросов» (что отключено и почему).

### 4.4 A/B-тесты креативов (нужно сделать)

- Триггер: cron каждые 7 дней или вручную через `/webhook/yadirect-abtest`.
- Логика:
  1. Выбрать в кампании активные объявления с показами ≥ 200.
  2. Выбрать «лидера» (CTR top-1) и «слабых» (CTR < 50% от лидера или CTR < 0.2%).
  3. Сгенерировать через Claude 2 новых варианта на замену каждому слабому, **сохраняя стиль лидера**.
  4. Прислать в TG: «Лидер: вариант 3 (1.4% CTR). Слабые: 1, 2. Заменить на новые [показать тексты]? ✅/❌»
  5. По подтверждению — `ads.suspend` слабых + `ads.add` новых через прокси.
- Промпт LLM:
  ```
  Ты опытный директолог-копирайтер. Вот объявления, которые хорошо работают:
  [LEADER_TEXTS]

  Вот объявления, которые работают плохо:
  [WEAK_TEXTS]

  Сгенерируй 2 НОВЫХ варианта на замену каждому слабому. Перенимай тон
  и структуру лидера, но избегай повторов. Каждый вариант: 5 заголовков
  до 56 символов, 3 описания до 81 символа. Верни JSON.
  ```

### 4.5 Триггер-наблюдатель (нужно сделать)

- Триггер: cron `*/30 * * * *` (каждые 30 минут).
- Что проверяет:
  - **Перерасход:** если расход за день > 90% дневного бюджета — пауза + алерт «бюджет почти кончился».
  - **Обвал CTR:** если за последние 60 минут CTR < 30% от среднего за 7 дней при показах > 50 — алерт.
  - **Резкий CPC:** если CPC за час > 200% от среднего недельного — алерт.
  - **Модерация:** если объявление отклонено модератором — алерт.
- Алерты идут в TG **с кнопкой автодействия** (например «приостановить кампанию»).

### 4.6 Цели в Метрике (нужно сделать на сайте + воркфлоу)

- На `studiache.ru` поставить счётчик Метрики (см. §6.1).
- Настроить цели:
  - `course_view` — просмотр страницы курса > 30 секунд.
  - `cta_click` — нажатие кнопки «Купить».
  - `payment_started` — открытие формы ЮKassa.
  - `payment_completed` — успешная оплата (через postback ЮKassa).
- Воркфлоу `24-metrika-sync.json` каждые 6 часов через [Reports API Метрики](https://yandex.ru/dev/metrika/doc/api2/api_v1/intro.html) тянет конверсии и сохраняет в static data, пометить как источник `direct.yandex` через UTM-метки `?utm_source=yandex&utm_medium=cpc&...`.
- Без этого CPA = неизвестен → весь L1/L2/L3 работает по показам/CTR, без оптимизации под продажи.

### 4.7 Управление через TG-команды

В оркестраторе добавить keyword-роутинг для команд:
- `дир статы` / `дир статы за неделю`
- `дир пауза <кампания>`
- `дир запуск <кампания>`
- `дир бюджет <кампания> <число>`
- `дир минус <слово> для всех`
- `дир ab` — запустить A/B-тест вручную

## 5. Нефункциональные требования

### 5.1 Безопасность бюджета

- **Жёсткий лимит:** агент НЕ может за один день увеличить дневной бюджет кампании более чем на 50% от текущего.
- **Жёсткий лимит:** агент НЕ может суммарно по аккаунту увеличить расход более чем на 100% за неделю.
- **Запрет на новые кампании без человека:** L3 не имеет права создавать кампанию.
- **Чёрный список действий L3:** `campaigns.add`, `campaigns.delete`, `raise_budget` > 10%, любое `ads.add`.

### 5.2 Безопасность доступа

- `YANDEX_DIRECT_TOKEN` хранится **только** в env сервера, никогда в репозитории.
- Webhook `/webhook/yadirect` доступен только из локальной сети n8n или с проверкой `X-Internal-Token` (env `YADIRECT_INTERNAL_TOKEN`).
- Все операции типа `campaigns.delete` требуют дополнительного подтверждения «введи 4-значный код, пришлю в TG».

### 5.3 Надёжность

- Retry 3×/1500мс на все вызовы Direct/Метрика.
- Если Direct в дауне — отчёт без рекомендаций, алерт «Direct API недоступен».
- Static data рекомендаций живёт 7 дней, потом GC.
- Все ошибки логируются в `02-error-handler` → админ-чат.

### 5.4 Лимиты L3 (автоматических действий)

L3 разрешён ТОЛЬКО для:
1. Добавления минус-слова, если ключевая фраза дала ≥ 3 клика и 0 конверсий за 7 дней при стоимости клика выше среднего по кампании.
2. Понижения ставки на ±10%, если CTR ключа < 0.2% при показах ≥ 100.
3. Паузы объявления, если 0 показов за 7 дней при бюджете > 0.

Все остальные изменения — только L2 (с подтверждением).

## 8. Ограничения и риски

| Риск | Вероятность | Митигация |
|---|---|---|
| Direct отклонит заявку API | средняя | Переподача с уточнённой формулировкой; временно работа через UI |
| LLM сгенерит плохие тексты объявлений | средняя | Подтверждение каждого нового объявления владельцем (L2) |
| Direct поменяет схему UnifiedCampaign | низкая | JSON-шаблоны кампаний хранятся отдельно, легко править |
| Бюджет потрачен на нецелевую аудиторию | средняя | Лимит дневного бюджета 300₽, недельный обзор, минус-слова |
| Метрика не отдаёт конверсии (cookies, AdBlock) | средняя | Дополнительно postback ЮKassa напрямую в n8n |
| Static data сбрасывается при рестарте | низкая (если volume есть) | Решается задачей #6 общего todo (Redis/persistent) |

## 9. Метрики успеха

Через 30 дней работы агента:

- **Время владельца на Директ ≤ 10 минут в день** (только просмотр сводки + 1–2 клика).
- **CPA на основном курсе** (видеомонтаж 4 900₽) — ≤ 1 500₽ (целевое 30% от чека).
- **ROAS** (выручка / расход) ≥ 1.5 за месяц.
- **Доля автоприменённых рекомендаций** ≥ 70% (агент попадает в цель часто).
- **Ноль перерасходов дневного бюджета** > 110%.
